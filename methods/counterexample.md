# Method: Counterexample

## Purpose

Test the boundary of a claim by looking for cases where it does not hold.

This method prevents overgeneralization and sharpens conceptual understanding.

## When to Use

- The user or source makes a broad claim.
- A concept seems too absolute.
- The system needs to test whether a distinction is real.
- The user is accepting an idea too quickly.

## When NOT to Use

- The user needs support or grounding before critique.
- The claim is already narrow and well-qualified.
- The task is memorization rather than evaluation.
- The counterexample would be irrelevant edge-case hunting.

## Inputs

- Claim or rule to test.
- Conditions under which the claim is supposed to hold.
- Candidate examples and edge cases.
- User's current confidence in the claim.

## Workflow

1. State the claim clearly.
2. Identify its implied scope.
3. Generate one plausible case that challenges the claim.
4. Ask whether the claim survives, needs revision, or should be rejected.
5. Refine the claim to include boundaries and conditions.

## Outputs

- A tested claim.
- Boundary conditions.
- A revised or more precise statement.
- Possible exceptions.

## Success Criteria

- The user sees the limits of the idea.
- The claim becomes more precise.
- The counterexample is fair and relevant.
- The user avoids either blind acceptance or shallow dismissal.

## Memory Integration

- Update Concept History with boundary conditions.
- Record corrected overgeneralizations in Misconception History.
- Add useful edge cases to the Idea Network.
- Preserve revised claims as memory anchors.

## Compatible Skills

- Deep Dive
- Synthesis
- Application
- Reflection
